# eat_what : 基于用户画像与问答引导的"今天吃什么"智能推荐微信小程序设计

## 主要功能界面

- 登录页面

  - 登录页面提供用户名+密码登录
  - 同时该页面提供一个注册按钮
  - 注册按钮进入注册页面（输入用户名和密码），保存到 DB 即可完成注册

- 界面模块设计

  - 底部呈现多个页面进入选项
    1. 首页面（今天吃什么页面）
    1. 个人历史记录页面
    1. 个人主页页面

- 首页面

  - 呈现两个入口：
    - "看看今天吃什么"：进入问答表单推荐流程
    - "浏览附近美食"：基于当前位置直接浏览附近商品列表
  - 问答表单会让用户记录其当前状态信息（问答选项前端会配置好）
    - 表单目的是设计一套交互式问答流程，用于捕捉用户的即时性、场景化需求。
    - **当前问答表单（共 8 题）**：
      1. **当前用餐时段**（必选）：早餐 / 午餐 / 下午加餐 / 晚餐 / 夜宵 / 随机/非正餐
      2. **用餐人数与场景**（必选）：单人简餐（快速解决） / 单人精致餐（慢食） / 双人约会/小聚 / 朋友小聚（3-5人） / 多人聚餐（6人以上） / 家庭日常用餐 / 家庭正式聚餐/宴请
      3. **本次用餐核心目的**（必选）：简单填饱肚子 / 日常家常餐 / 享受美味大餐 / 解馋/满足欲望
      4. **本次选择偏好**（必选）：只吃熟悉口味 / 愿意尝试新口味 / 优先热门爆款 / 无所谓，好吃就行
      5. **用餐形态**（必选）：正餐主食 / 快餐简餐 / 小吃点心 / 甜品/饮品 / 轻食
      6. **预算范围**（多选）：20元以下 / 20-50元 / 50-100元 / 100元以上
      7. **意向菜系/品类**（多选，可选）：烧烤烤肉 / 奶茶果汁 / 炸鸡炸串 / 鸭脖卤味 / 特色小吃 / 米粉面条 / 快餐便当 / 汉堡薯条 / 粥食点心 / 地方菜系 / 麻辣烫冒菜 / 饺子馄饨
      8. **六维口味强度偏好**（可选，合并展示）：辣度 / 麻度 / 酸度 / 甜度 / 咸度 / 油度（0~5滑块，不选表示无特别偏好）
    - 向量构建逻辑：后端将上述问答字段与用户画像信息拼接为结构化语义文本，叠加衍生特征（饱腹度、浓郁度、社交属性）后，经 sentence-transformers 编码为 768 维用户向量，传递给推荐引擎用于语义召回。
  - 表单提交后，从后台获取相应的推荐菜品数据，形成流式列表反馈给用户
  - 问卷页面包含加载态与失败重试兜底；当题目接口超时或异常时展示错误提示与“重新加载”按钮，避免空白页
  - 对于呈现的推荐选项，用户可以对推荐菜品选择就决定是你了，后台记录用户选择（可在历史记录查看）并快速返回首页
    - 后台还会保存用户当前选择形成的画像，供后面决策进行优化用户偏好： 用户即时画像（那一刻的问答结果），推荐列表中的前 N 项（正负样本数据），时间戳（方便未来训练模型）

- 历史记录页 ：用户可以回顾历史选择，并查看每条记录对应菜品详情（商家/菜系/口味/描述等）

- 个人主页页面
  - 提供跳转按钮 1 ：前往编辑用户信息的页面
    - 录入并管理个人静态信息
      - **具体信息参数明细：**
        - **基础生理信息**：年龄、性别、身高、体重、活动系数（久坐1.2、轻度活动1.375、中度1.55、高强度1.725）。
        - **健康目标**：减脂 / 增肌 / 维持现状。
        - **忌口与饮食限制**（多选）：清真（不吃猪肉）、素食、花生过敏、海鲜过敏、乳糖不耐、不吃香菜、不吃葱/蒜、不吃辣、不吃油腻。
        - **口味偏好**：酸/甜/苦/辣/咸度等级（0~5评分）。
        - **菜系偏好**（多选）：烧烤烤肉 / 奶茶果汁 / 炸鸡炸串 / 鸭脖卤味 / 特色小吃 / 米粉面条 / 快餐便当 / 汉堡薯条 / 粥食点心 / 地方菜系 / 麻辣烫冒菜 / 饺子馄饨。
  - 提交商店菜品信息的按钮，会进入一个表单
    - 需要指定菜品名字，菜系，口味，商店位置等一系列信息，提交成功后，后台根据相应信息生成菜品的权重向量
    - 文本（菜名、口味描述）→ 使用 `sentence-transformers` 本地模型 `shibing624/text2vec-base-chinese` 生成向量

## 推荐系统

- 根据用户上传数据生成用户当前的用户画像向量
- 从数据库获取已审核的菜品，计算每个菜品的推荐得分
- 最终返回推荐列表数据

- **智能推荐模块设计（核心推荐策略）**：
  融合用户画像向量与即时问答特征后生成目标向量，进行以下三段式筛选与排序：

  1. 硬性过滤（Rule-based）：剔除超出预算的菜品、未审核菜品、命中用户忌口的菜品。
  2. 向量召回（Vector Recall）：用户向量与菜品向量分别经菜系/口味/形态扩展描述后编码，使用 `cosine_similarity` 召回候选。
  3. 综合重排（Ranking）：
     - 最终得分 = 0.16×语义相似度 + 0.07×距离惩罚 + 0.12×问卷菜系匹配 + 0.08×画像菜系补偿 + 0.08×意图命中 + 0.11×口味兼容度 + 0.09×用餐形态兼容度 + 0.08×用餐语境兼容度 + 0.12×忌口兼容度 + 0.06×夜宵/小吃场景兼容度 + 0.06×子类型精确度 + 0.10×画像口味兼容度 + 0.04×历史偏好匹配 + 0.01×历史去重 + 0.02×预算匹配度
     - **口味兼容度**：用户口味强度（辣/麻/油等）与菜品标签的匹配程度
     - **画像口味兼容度**：用户画像中长期口味偏好与菜品标签的匹配程度
     - **忌口兼容度**：用户画像忌口（不吃辣/不吃油腻等）与菜品内容的冲突惩罚
     - **菜系匹配**：用户选择的意向菜系（问卷或画像）与菜品菜系一致时加分
     - **历史偏好匹配**：当前问卷与历史问卷相似度越高，历史选择的参考权重越大
     - **夜宵/小吃场景兼容度**：夜宵/小吃场景下鼓励烧烤卤味炸串，抑制主食套餐披萨

- **商家类型解析模块**：
  从商家名称自动提取主营品类（如"杜小姐在西北·肉夹馍·凉皮·油泼面"解析出 [肉夹馍, 凉皮, 油泼面]），映射到12大分类并推断口味倾向，用于增强菜品向量化质量。

- **食谱/餐厅信息管理模块**：
  用于商家端或后台管理员录入基础数据。字段包括：菜品名称（文本）、宣传图片（URL 列表）、主要食材（拆分词库形式录入，如：猪肉/辣椒/大蒜）、口味分类枚举（酸/甜/辣/咸/清淡/苦）、商家主营品类（JSON）、商家口味上下文（JSON）、商家场景（字符串）、价格（定值浮点数）。
  - 店铺需要审核通过后才展示
  - 菜品提交后需要审核通过后才展示
  - 支持按商家查看其所有菜品

## 数据库数据 (核心实体预设设计明确)

    - **用户表 (users)**:
        - 属性：用户ID (String, 唯一索引), 密码Hash, 注册时间, 账号状态 (正常/封禁), 角色ID.
    - **用户画像表 (user_profiles)**:
        - 属性：画像ID (PK), 用户ID (FK, 一对一), 年龄 (Int), 性别 (Int), 身高 (Float), 体重 (Float), 活动系数 (Float), 健康目标 (String), 口味偏好 (JSON), 菜系偏好 (JSON), 忌口 (JSON).
    - **菜品表 (dishes)**:
        - 属性：菜品ID (PK), 菜品名称, 所属店铺ID (FK), 菜系 (String), 口味标签 (JSON), 口味分数 (JSON), 口味详细标签 (JSON), 价格 (Decimal), 主要食材 (JSON), 宣传图片 (JSON), 菜品描述 (Text).
        - 商家关联字段：商家主营品类 (JSON), 商家口味上下文 (JSON), 商家场景 (String).
        - 向量字段：预计算的 768 维菜品向量 (JSON).
        - 审核状态：is_approved (Boolean).
    - **店铺表 (shops)**:
        - 属性：店铺ID (PK), 店铺名称, 地址, 城市, 经纬度, 审核状态.
    - **历史选择表 (history_records)**:
        - 属性：记录ID (PK), 用户ID (FK), 推荐批次ID, 选择的菜品ID (FK), 就餐时间, 问答快照 (JSON), 是否采纳首条推荐 (Boolean).
    - **交互日志表 (interaction_logs)**:
        - 属性：日志ID (PK), 用户ID, 推荐批次ID, 推荐菜品ID列表 (JSON), 交互时间, 点击菜品ID, 结果 (正/负样本), 问答快照 (JSON).

## 实现步骤

1. **前端开发**：
   - 使用微信小程序原生框架实现页面布局与交互逻辑。
   - 页面包括：登录、注册、问答表单、推荐结果、菜品详情、历史记录、个人主页、编辑画像、附近美食（菜品/商家）、商家详情、店铺录入、菜品录入等。

2. **后端开发**：
   - 使用 FastAPI 框架开发后端服务，提供 RESTful API。
   - 路由模块：auth_router（认证）、profile_router（画像）、dish_router（菜品管理）、recommend_router（推荐）、history_router（历史记录）、nlp_router（向量化服务）。

3. **数据库设计**：
   - 使用 MySQL 创建用户表、用户画像表、菜品表、店铺表、历史记录表、交互日志表、角色表等。
   - 使用 SQLAlchemy ORM 的 `Base.metadata.create_all` 自动建表并维护当前模型对应结构。

4. **AI 模型集成**：
   - 使用 `sentence-transformers` 本地模型 `shibing624/text2vec-base-chinese` 生成菜品和用户画像向量。
   - 推荐算法采用向量相似度计算 + 规则过滤 + 综合重排。

## 数据导入与向量化

### 从爬虫数据导入菜品

将饿了么爬虫数据转换为向量并导入数据库：

```powershell
# 直接导入数据库（清空旧数据）
python backend/scripts/prepare_eleme_seed.py `
  --input get_data/.generated `
  --import-db `
  --clear-before-import
```

### 提取商家名称（用于分析分类）

```powershell
python get_data/extract_shop_names.py

# 输出文件：get_data/.generated/shop_names.txt
```

### 重新生成向量（向量化逻辑更新后）

如果 vectorize.py 的 `build_dish_text` 或 `generate_user_vector` 有更新：

```powershell
# 方案一：重新向量化数据库中已有菜品（推荐）
python backend/scripts/revectorize_dishes.py

# 方案二：删除旧数据后重新从爬虫数据导入
Remove-Item backend/.generated/eleme_seed_payload.json
python backend/scripts/prepare_eleme_seed.py `
  --input get_data/.generated `
  --import-db `
  --clear-before-import
```

## 附近美食功能

### 周边菜品
- 按当前位置展示半径 5km 内的已审核菜品
- 展示菜品图片、名称、价格、距离
- 点击跳转菜品详情

### 周边商家
- 按当前位置展示半径 5km 内的已审核商家
- 展示商家名称、地址、菜品数量、距离
- 点击跳转商家详情页，展示该商家的所有菜品
- API: `GET /api/recommend/nearby-shops`
- API: `GET /api/recommend/shop/{shop_id}/dishes`

## 审核机制

- **店铺审核**：新提交的店铺默认 `is_approved=false`，需管理员审核后才展示
- **菜品审核**：新提交的菜品默认 `is_approved=false`，需管理员审核后才展示
- 管理员可在管理后台进行审核操作
