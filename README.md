# eat_what : 基于用户画像与问答引导的“今天吃什么”智能推荐微信小程序设计

## 主要功能界面

- 登录页面

  - 登录页面提供用户名+密码登录
  - 同时该页面提供一个注册按钮
  - 注册按钮进入注册页面（输入用户名和密码），保存到 DB 即可完成注册

- 界面模块设计

  - 底部呈现多个页面进入选项
    1. 首页面（今天吃什么页面）
    1. 个人历史记录页面
    1. 个人主页页面

- 首页面

  - 呈现两个入口：
    - “看看今天吃什么”：进入问答表单推荐流程
    - “浏览附近美食”：基于当前位置直接浏览附近商品列表
  - 问答表单会让用户记录其当前状态信息（问答选项前端会配置好）
    - 表单目的是设计一套交互式问答流程，用于捕捉用户的即时性、场景化需求。
      - **已补充的常见场景化问题库：**
        - **就餐时段**：“早餐 / 午餐 / 晚餐 / 夜宵 / 下午茶？”
        - **口味偏好**：“今天想吃点清淡的还是重口的？（清爽、麻辣、酸甜、咸鲜）”
        - **就餐场景**：“一个人随便吃吃 / 约会 / 同事聚餐 / 家庭聚餐？”
        - **就餐形式**：“叫外卖 / 堂食 / 随买随走的快餐？”
        - **预算范围**：“穷鬼套餐（20 元以下）/ 基础管饱（20-50 元）/ 稍微小资（50-100 元）/ 吃顿好的（100 元以上）？”
        - **特殊状态**：“需要解压（推荐高热量甜品/炸鸡）/ 正在减脂 / 胃不舒服（推荐流食/养胃）？”
    - 动态问答逻辑：根据前面问题的回答动态调整后续问题
      - 例如：如果用户说“今天想吃清淡”，后续就可以问“要不要推荐汤类？”；如果预算很低，可以减少正餐、推荐快餐或外卖。
      - 权重积累：每次回答一个问题，就给某个标签权重加分（如“辣度+0.8”、“汤类+0.6”）。最终把这些临时权重叠加到用户画像向量中，生成“即时画像”。后续会传递给推荐系统用于推荐
  - 表单提交后，从后台获取相应的推荐菜品数据，形成流式列表反馈给用户
  - 对于呈现的推荐选项，用户可以对推荐菜品选择就决定是你了，后台记录用户选择（可在历史记录查看）并退出回到起始页面
    - 后台还会保存用户当前选择形成的画像，供后面决策进行优化用户偏好： 用户即时画像（那一刻的问答结果），推荐列表中的前 N 项（正负样本数据），时间戳（方便未来训练模型）

- 历史记录页 ：用户可以回顾历史选择，并查看每条记录对应菜品详情（商家/菜系/口味/描述等）

- 个人主页页面
  - 提供跳转按钮 1 ：前往编辑用户信息的页面
    - 录入并管理个人静态信息
      - **具体信息参数明细：**
        - **基础生理信息**：年龄、性别、身高、体重、活动系数（久坐、轻度活动、高强度等，用于估算基础代谢 TDEE）。
        - **健康目标**：减脂 / 增肌 / 维持现状 / 养生（低油低盐）。
        - **强制性限制（硬性过滤条件）**：宗教禁忌（如清真）、素食者（纯素/蛋奶素）、常见过敏原（花生、海鲜、乳糖、麸质等）。
        - **动态/长效偏好**：口味偏好（酸/甜/苦/辣/咸度等级）、菜系偏好（川湘菜、江浙菜、日韩料理、西餐点心等）、忌口食材（不吃香菜、葱、蒜、内脏等）。
  - 提交商店菜品信息的按钮，会进入一个表单
    - 需要指定菜品名字，菜系，口味，商店位置等一系列信息，提交成功后，后台根据相应信息生成菜品的权重向量
    - 文本（菜名、口味描述） → 使用 `sentence-transformers` 本地模型 `shibing624/text2vec-base-chinese` 生成向量；
    - 数值（价格、热量） → 归一化后拼接到文本向量后面;
    - 标签（菜系、口味） → 进行编码并并入推荐特征。

## 推荐系统

- 根据用户上传数据生成用户当前的用户画像向量
- 从数据库获取同城且已审核的菜品，计算每个菜品的推荐得分（距离、菜品向量与用户向量 match 值、历史平滑）
- 最终返回推荐列表数据

- **用户画像模块设计**：
  用户录入并管理个人静态与动态信息。具体字段要求如下：

  - 基础生理信息：年龄（整型）、性别（男/女）、身高（cm）、体重（kg）、活动系数（久坐=1.2、轻度=1.375、中度=1.55、高强度=1.725）。
  - 健康目标（单选）：减脂、增肌、维持现状、养生饮食。
  - 饮食禁忌与过敏原（多选）：清真、纯素食、蛋奶素食；花生、海鲜、乳糖、麸质。
  - 动态偏好：口味偏好等级（酸/甜/苦/辣/咸，各为 0-5 级评分）、菜系偏好（川湘菜/江浙菜/日韩料理/西餐/快餐便当，多选列表）、忌口食材（香菜/葱/蒜/姜/动物内脏，多选列表）。

- **问答引导模块设计（增强版）**：
  采用“时间→场景→目的→口味→菜系→强度→约束”递进链路，形成可直接向量化的即时画像。
  当前实现包含：
  1. 时段：早餐/午餐/下午加餐/晚餐/夜宵/凌晨加餐/随机。
  2. 场景：单人简餐、单人精致、双人约会、朋友小聚、多人聚餐、家庭日常、家庭宴请。
  3. 目标：填饱、家常、大餐、解馋、轻食、暖胃。
  4. 选择偏好：保守、尝新、热门、随意。
  5. 用餐形态：正餐/快餐/小吃/甜品饮品/轻食代餐。
  6. 核心口味：重口、清淡、酸甜、汤粥、肉食、奶甜、酱卤、孜然烧烤。
  7. 意向菜系（最多2项）+ 食材偏好 + 忌口多选。
  8. 强度滑条：辣/麻/酸/甜/咸/油（0~5）。
  9. 口感、温度、特殊需求（文本）。

  生成特征：
  - 类别特征（场景、目的、菜系、形态等）
  - 连续特征（六维强度）
  - 约束特征（忌口、特殊需求）
  - 衍生特征（饱腹度、浓郁度、社交属性）
- **智能推荐模块设计（核心推荐策略）**：
  融合用户画像向量与即时问答特征后生成目标向量，进行以下三段式筛选与排序：

  1. 硬性过滤（Rule-based）：剔除超出预算、不符合就餐形式、命中有害过敏原或饮食禁忌的菜谱/商家分类。
  2. 向量召回（Vector Recall）：在后端使用 `sklearn.metrics.pairwise.cosine_similarity` 对用户向量与候选菜品向量计算相似度并召回候选。
  3. 综合重排（Ranking）：最终得分 = 向量相似度得分(0.6 权重) + 距离惩罚项(距离越远得分越低，0.3 权重) + 历史偏好平滑系数(重复推荐降权，0.1 权重)。

- **食谱/餐厅信息管理模块**：
  用于商家端或后台管理员录入基础数据。限制录入字段如下：菜品名称（文本）、宣传图片（URL 列表）、主要食材（拆分词库形式录入，如：猪肉/辣椒/大蒜）、营养成分（卡路里 kcal，蛋白质 g，脂肪 g，碳水 g）、口味分类枚举（酸/甜/辣/咸/清淡）、价格（定值浮点数）。

## 数据库数据 (核心实体预设设计明确)

    - **用户表 (Users)**:
        - 属性：用户ID (字符串，唯一索引), 密码Hash, 注册时间, 账号状态 (正常/封禁)。
    - **用户画像表 (User_Profiles)**:
        - 属性：画像ID (PK), 用户ID (FK), 年龄 (Int), 性别 (TinyInt), 身高 (Float), 体重 (Float), 活动系数 (Float), 健康目标 (Varchar), 禁忌过敏原枚举 (JSON), 口味偏好向量 (JSON/Vector), 忌口字典 (JSON)。
    - **菜品及店铺表 (Dishes)**:
        - 属性：菜品ID (PK), 菜品名称 (Varchar), 所属店铺ID (FK), 地理位置 (经纬度), 菜系 (Varchar), 口味标签枚举 (JSON), 价格 (Decimal), 卡路里 (Float), 宏量营养素组 (JSON), 主要食材成分 (JSON)。
        - 向量字段：预计算好的菜品高维嵌入向量（Dish Vector，768维Float Array）。
    - **历史选择表 (History_Records)**:
        - 属性：记录ID (PK), 用户ID (FK), 推荐批次标识ID, 最终选择菜品ID (FK), 就餐时间 (Datetime), 即时问答特征JSON (当前问答快照), 是否采纳系统第一推荐 (Boolean)。
    - **即时交互日志表 (Interaction_Logs)**:
        - 属性：日志ID (PK), 用户ID, 推荐菜品ID列表 (JSON/Array), 用户发生的点击交互时间戳, 判定结果 (1正样本/0负样本)，用于在线学习模型的回流。

## 实现步骤

1. **前端开发**：
   - 使用微信小程序原生框架实现页面布局与交互逻辑。
   - 页面包括：登录、注册、问答表单、历史记录、个人主页等。

2. **后端开发**：
   - 使用 FastAPI 框架开发后端服务，提供 RESTful API。
   - 实现用户管理、问答数据处理、推荐系统逻辑、历史记录存储等功能。

3. **数据库设计**：
   - 使用 MySQL 创建用户表、用户画像表、菜品表、历史记录表等。
   - 使用 SQLAlchemy ORM 的 `Base.metadata.create_all` 自动建表并维护当前模型对应结构。

4. **AI 模型集成**：
   - 使用 `sentence-transformers` 本地模型 `shibing624/text2vec-base-chinese` 生成菜品和用户画像向量。
   - 推荐算法采用向量相似度计算 + 规则过滤 + 综合重排。
